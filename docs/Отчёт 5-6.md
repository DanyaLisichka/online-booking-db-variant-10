# Отчёт по практической работе

## Проектирование и реализация пользовательского интерфейса системы онлайн-записи

Целью работы являлась разработка веб-интерфейса для системы онлайн-записи к психологам. В ходе выполнения были реализованы CRUD-интерфейсы для справочников, а также бизнес-операции, связанные с созданием и управлением записями на консультации.

В работе использовались PHP, MySQL, PDO, HTML, CSS, Bootstrap 5 и JavaScript.

# Часть 1. Реализация справочников

В первой части работы был разработан интерфейс управления справочными данными. Были реализованы следующие справочники:

- клиенты;
- методики;
- психологи;
- записи на консультацию.

В каждом справочнике реализованы основные CRUD-операции:
- добавление новой записи;
- просмотр списка записей;
- редактирование существующей записи;
- удаление с подтверждением.

Для удобства пользователя на страницах списка были добавлены поиск, сортировка по столбцам и пагинация. Количество записей на странице ограничено.

## Архитектура приложения

Приложение реализовано с использованием упрощённого MVC-подхода.
В качестве модели используются репозитории:
```
ClientRepository
MethodRepository
PsychologistRepository
AppointmentRepository
ServiceRepository
ReportRepository
```
В репозиториях находятся SQL-запросы и методы доступа к данным:
```
findAll()
findById()
create()
update()
delete()
findPaginated()
countAll()
```
Контроллеры принимают запрос пользователя, вызывают нужные методы репозиториев и передают данные в представления.

Примеры контроллеров:
```
ClientController
MethodController
PsychologistController
AppointmentController
BookingController
ReportController
```
Представления находятся в папке views. Они отвечают только за HTML-разметку и вывод данных. SQL-запросы в представлениях не используются.

## Маршрутизация

Маршрутизация реализована через параметры entity и action в URL.

Примеры маршрутов:
```
index.php?entity=client&action=list
index.php?entity=psychologist&action=create
index.php?entity=appointment&action=edit&id=3
index.php?entity=booking&action=create
index.php?entity=report&action=index
```
Главный файл index.php определяет нужный контроллер и вызывает соответствующий метод.
  
## Валидация и безопасность

В приложении реализована серверная валидация данных. Проверяются обязательные поля, корректность email, телефона, даты и допустимость выбранных значений.
При ошибках форма повторно отображается с уже введёнными данными, а ошибочные поля подсвечиваются.

Для защиты от XSS все выводимые данные экранируются с помощью:
```
htmlspecialchars()
```
Для защиты форм используется CSRF-токен. Токен генерируется при открытии формы и проверяется при отправке POST-запроса.

Также все SQL-запросы выполняются через PDO prepared statements, что защищает приложение от SQL-инъекций.

## Проверка связей при удалении

При удалении связанных сущностей выполняется проверка наличия зависимых записей.

Например:

- нельзя удалить психолога, если у него есть записи на консультации;
- нельзя удалить методику, если она связана с психологами;
- нельзя удалить запись, если к ней привязаны заметки психолога.

Если удаление невозможно, пользователю выводится flash-сообщение с описанием причины.

# Часть 2. Бизнес-операция бронирования

Во второй части работы была реализована основная бизнес-операция системы — бронирование консультации.

Пользователь выбирает:

1. услугу;
2. психолога;
3. дату;
4. доступный временной слот;
5. клиента;
6. тип консультации.

После выбора услуги список психологов загружается динамически через AJAX. После выбора психолога и даты динамически загружаются свободные временные слоты.

Для AJAX-запросов были созданы отдельные endpoint-файлы:

```
ajax/get_psychologists.php
ajax/get_available_slots.php
```
Endpoint возвращает данные в формате JSON.

## Работа со слотами

Свободные временные слоты рассчитываются на сервере. Рабочий день задан с 09:00 до 18:00, обеденный перерыв — с 13:00 до 14:00.

При расчёте учитываются уже существующие записи психолога на выбранную дату. Занятые слоты исключаются из списка доступных.

Перед созданием записи сервер повторно проверяет, что выбранный слот ещё свободен. Это защищает от ситуации, когда два пользователя одновременно пытаются записаться на одно и то же время.

## Отчёты

На странице выводятся следующие отчёты:

1. количество записей и выручка по дням;
2. загруженность психологов;
3. выручка по услугам.

Также реализован экспорт отчёта в CSV-файл.

# Контрольные вопросы
## Какие статусы записи есть в вашей системе? Какие переходы между статусами разрешены?

В системе используются четыре статуса записи на консультацию:
```
pending — ожидает подтверждения
confirmed — подтверждена
cancelled — отменена
completed — завершена
```
При создании новой записи ей автоматически присваивается статус pending, то есть запись ожидает подтверждения.

Разрешённые переходы между статусами:
```
pending → confirmed
pending → cancelled
confirmed → completed
confirmed → cancelled
```
То есть запись сначала создаётся как ожидающая. После этого её можно подтвердить или отменить. Подтверждённую запись можно завершить после проведения консультации либо отменить, если консультация не состоялась.

Для завершённых записей изменение статуса запрещено. Это сделано для сохранения корректной истории консультаций: если запись уже имеет статус completed, её нельзя снова перевести в ожидающую, подтверждённую или отменённую. Отменённая запись также считается конечным состоянием, так как её временной слот освобождается для других бронирований.
